### Design intent

- Primary purpose: Guide provider-specific cloud authorization.
- Primary actors: Clara the Cloud Collector
- Primary job: Authorize Dropbox, Google Drive, iCloud Drive, OneDrive, or advanced WebDAV.
- Principle served: Cloud providers are first-class.
- Entry points: Sources selected cloud provider, Source Repair Reauthorize provider.
- Success outcome: Provider auth succeeds and folder browsing can begin.
- Content priority: Primary: selected provider, auth status, Start/Retry authorization. Secondary: account and permission details.
- Critical variants: provider_selected, auth_pending, auth_success, auth_error
- User state of mind: Focused on authorize dropbox, google drive, icloud drive, onedrive, or advanced webdav.
- Usage context: Android TV, 10-foot viewing distance, D-pad remote, home network/cloud source variability, overscan-safe layout.
- Data range: Zero, typical, maximum, long text, missing artwork/metadata, and source-permission variants must render without overlap.
- Anti-goals: Do not expose raw credentials, OAuth tokens, provider file IDs, secret URLs, stack traces, live folder traversal on cached browsing screens, phone navigation, or hover/touch-only controls.
- Fidelity target: Mid-fi
- Exit paths: Folder Picker, Source Repair, Sources.

### Screen generation prompt

> Target tool: Other

Design an Android TV Provider Authorization screen for Clara to authorize a selected cloud provider. The selected provider must be visually clear, auth pending must work when she switches to a phone or browser, and success/error states must be distinct. Keep first-class provider language visible and avoid raw OAuth jargon, tokens, URLs, or provider IDs.

### Acceptance criteria

- [ ] Clara can tell which provider she is authorizing.
- [ ] Auth pending, success, and error states are visually distinct.
- [ ] Retry and Cancel paths are available without losing the selected provider.
- [ ] Offline/auth-denied messages are specific and recovery-oriented.
- [ ] No raw tokens, URLs, or provider IDs are visible.
- [ ] Focus, disabled, loading, success, and error feedback are visually defined for consequential actions.
- [ ] All content uses realistic mock data and no placeholder text.
