### Design intent

- Primary purpose: Control audiobook playback and preserve progress.
- Primary actors: Leo the Living Room Listener; Playback Engine
- Primary job: Listen, pause, seek, change speed, and recover from playback trouble.
- Principle served: Resume is sacred.
- Entry points: Resume from Home, Resume/Start from Book Detail, media session return.
- Success outcome: Audio plays from selected position and progress is saved reliably.
- Content priority: Primary: playback state, title, progress, play/pause, seek, speed. Secondary: source health and repair.
- Critical variants: buffering, playing, paused, playback_error
- User state of mind: Focused on listen, pause, seek, change speed, and recover from playback trouble.
- Usage context: Android TV, 10-foot viewing distance, D-pad remote, home network/cloud source variability, overscan-safe layout.
- Data range: Zero, typical, maximum, long text, missing artwork/metadata, and source-permission variants must render without overlap.
- Anti-goals: Do not expose raw credentials, OAuth tokens, provider file IDs, secret URLs, stack traces, live folder traversal on cached browsing screens, phone navigation, or hover/touch-only controls.
- Fidelity target: Mid-fi
- Exit paths: Home, Book Detail, Library, Source Repair, Android TV dashboard.

### Screen generation prompt

> Target tool: Other

Design an Android TV Now Playing screen for Leo to control a long audiobook from the couch. Show book/chapter context, saved position, timeline, play/pause, skip back/forward, speed, buffering, paused, playing, and playback-error states. Progress preservation is the trust signal: source failures must show saved position, Retry playback, and Repair source without exposing raw URLs or provider IDs.

### Acceptance criteria

- [ ] Leo can control playback with D-pad focus visible on every control.
- [ ] Playing, paused, buffering, and playback-error states are visually distinct.
- [ ] Saved position remains visible during error and repair entry.
- [ ] Speed and seek controls do not crowd the primary play/pause action.
- [ ] Back/dashboard behavior preserves playback context.
- [ ] Focus, disabled, loading, success, and error feedback are visually defined for consequential actions.
- [ ] All content uses realistic mock data and no placeholder text.
