### Design intent

- Primary purpose: Explain source failures and route to the right recovery action.
- Primary actors: Leo the Living Room Listener; Sam the Server Steward; Clara the Cloud Collector
- Primary job: Retry, repair credentials/auth, or safely continue with cached content.
- Principle served: Source problems need repair paths.
- Entry points: Home source unavailable, Now Playing playback error, Sync Progress partial/error, Book Detail source unavailable, Library stale source.
- Success outcome: The source recovers or the user has a clear next step while cached context remains available.
- Content priority: Primary: failed source, failure category, recommended recovery. Secondary: cached availability and origin return.
- Critical variants: source_unavailable, reauthorization_required, credentials_required, repair_complete
- User state of mind: Focused on retry, repair credentials/auth, or safely continue with cached content.
- Usage context: Android TV, 10-foot viewing distance, D-pad remote, home network/cloud source variability, overscan-safe layout.
- Data range: Zero, typical, maximum, long text, missing artwork/metadata, and source-permission variants must render without overlap.
- Anti-goals: Do not expose raw credentials, OAuth tokens, provider file IDs, secret URLs, stack traces, live folder traversal on cached browsing screens, phone navigation, or hover/touch-only controls.
- Fidelity target: Mid-fi
- Exit paths: Now Playing, Library, Home, SMB Setup, Provider Authorization.

### Screen generation prompt

> Target tool: Other

Design an Android TV Source Repair screen that explains why an audiobook source cannot sync or stream and gives the correct recovery action. Distinguish network/source unavailable, reauthorization required, SMB credential repair, and repair-complete states, preserve cached library and playback context, and avoid raw technical errors. Include Retry source, provider reauthorization or SMB credential repair when applicable, and View cached library when local metadata is available.

### Acceptance criteria

- [ ] Retry has loading, success, and repeated-error treatment.

- [ ] Repair-complete state returns to the originating task.
- [ ] Focus, disabled, loading, success, and error feedback are visually defined for consequential actions.
- [ ] All content uses realistic mock data and no placeholder text.
