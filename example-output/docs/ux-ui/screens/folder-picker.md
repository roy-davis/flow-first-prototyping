### Design intent

- Primary purpose: Let the source owner choose the audiobook root folder.
- Primary actors: Sam the Server Steward; Clara the Cloud Collector
- Primary job: Browse folders from the validated source and select the indexing root.
- Principle served: Cache first, never wait on folders.
- Entry points: SMB Setup validation success, Provider Authorization success.
- Success outcome: Selected folder starts background indexing.
- Content priority: Primary: source identity, folder path, folder list, Select this folder. Secondary: permissions/capabilities.
- Critical variants: folder_browsing, permission_blocked, empty_folder
- User state of mind: Focused on browse folders from the validated source and select the indexing root.
- Usage context: Android TV, 10-foot viewing distance, D-pad remote, home network/cloud source variability, overscan-safe layout.
- Data range: Zero, typical, maximum, long text, missing artwork/metadata, and source-permission variants must render without overlap.
- Anti-goals: Do not expose raw credentials, OAuth tokens, provider file IDs, secret URLs, stack traces, live folder traversal on cached browsing screens, phone navigation, or hover/touch-only controls.
- Fidelity target: Mid-fi
- Exit paths: Sync Progress, SMB Setup, Provider Authorization.

### Screen generation prompt

> Target tool: Other

Design an Android TV Folder Picker for Sam or Clara to choose the audiobook root folder from a validated SMB or cloud source. Show source identity, the current path, focusable folder rows, readable status, permission/capability warnings, and a stable Select this folder action. Render folder browsing, permission-blocked, and empty-folder states without becoming a generic file manager.

### Acceptance criteria

- [ ] The source owner can identify the current source and folder path.
- [ ] Select this folder is disabled or blocked for unreadable folders.
- [ ] Permission-blocked and empty-folder states give clear recovery.
- [ ] Long paths and folder names remain readable without overlapping actions.
- [ ] Back and Go up behavior are distinct.
- [ ] Focus, disabled, loading, success, and error feedback are visually defined for consequential actions.
- [ ] All content uses realistic mock data and no placeholder text.
