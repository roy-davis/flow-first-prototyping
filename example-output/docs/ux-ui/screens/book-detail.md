### Design intent

- Primary purpose: Show audiobook context and let the listener start/resume or add a discovered book to the listening library.
- Primary actors: Leo the Living Room Listener
- Primary job: Confirm a book and choose where playback begins.
- Principle served: Resume is sacred.
- Entry points: Library select book.
- Success outcome: Leo starts/resumes playback or understands source trouble.
- Content priority: Primary: title, author, progress, Resume/Start. Secondary: chapters. Tertiary: source metadata.
- Critical variants: ready, source_unavailable, no_chapters
- User state of mind: Focused on confirm a book and choose where playback begins.
- Usage context: Android TV, 10-foot viewing distance, D-pad remote, home network/cloud source variability, overscan-safe layout.
- Data range: Zero, typical, maximum, long text, missing artwork/metadata, and source-permission variants must render without overlap.
- Anti-goals: Do not expose raw credentials, OAuth tokens, provider file IDs, secret URLs, stack traces, live folder traversal on cached browsing screens, phone navigation, or hover/touch-only controls.
- Fidelity target: Mid-fi
- Exit paths: Now Playing, Library, Source Repair.

### Screen generation prompt

> Target tool: Other

Design an Android TV Book Detail screen for Leo to confirm an audiobook and start or resume playback, or to add a discovered audiobook to the listening library. Show title, author, saved progress or discovered status, source health, and a scrollable chapter list while keeping the primary action visually stable. Render ready, discovered-not-added, added, source-unavailable, and no-chapters states; source trouble should disable playback actions and show Repair source without hiding cached book context.

### Acceptance criteria

- [ ] Leo can start or resume the selected book without scanning technical metadata.
- [ ] Source-unavailable state keeps cached book context visible.
- [ ] Chapter list handles dense data without hiding the primary action.

- [ ] Raw provider IDs or paths are not primary UI copy.
- [ ] Focus, disabled, loading, success, and error feedback are visually defined for consequential actions.
- [ ] All content uses realistic mock data and no placeholder text.
