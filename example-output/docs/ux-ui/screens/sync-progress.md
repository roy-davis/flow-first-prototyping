### Design intent

- Primary purpose: Show background indexing progress and outcomes.
- Primary actors: Sam the Server Steward; Clara the Cloud Collector; Sync Worker
- Primary job: Confirm the selected source is being indexed into local metadata.
- Principle served: Cache first, never wait on folders.
- Entry points: Folder Picker Select this folder.
- Success outcome: The user can open Library when enough metadata is indexed or sync completes.
- Content priority: Primary: source name, status, progress, View library. Secondary: counts, skipped items, current path.
- Critical variants: queued, indexing, complete, partial_error
- User state of mind: Focused on confirm the selected source is being indexed into local metadata.
- Usage context: Android TV, 10-foot viewing distance, D-pad remote, home network/cloud source variability, overscan-safe layout.
- Data range: Zero, typical, maximum, long text, missing artwork/metadata, and source-permission variants must render without overlap.
- Anti-goals: Do not expose raw credentials, OAuth tokens, provider file IDs, secret URLs, stack traces, live folder traversal on cached browsing screens, phone navigation, or hover/touch-only controls.
- Fidelity target: Mid-fi
- Exit paths: Library, Source Repair, Back.

### Screen generation prompt

> Target tool: Other

Design an Android TV Sync Progress screen showing the Sync Worker indexing a selected audiobook source into local metadata. Show source identity, progress, counts, current item, and clear queued, indexing, complete, and partial-error states. The user can leave sync running, review discovered books when sync has produced a result, retry, or review source issues without blocking the whole app.

### Acceptance criteria

- [ ] Sync status is visible and understandable from 10 feet.
- [ ] Complete state has a clear Review discovered books action.
- [ ] Partial errors preserve usable indexed results and expose recovery.
- [ ] Retry cannot be double-triggered while sync is running.
- [ ] Long current item/path does not overlap actions.
- [ ] Focus, disabled, loading, success, and error feedback are visually defined for consequential actions.
- [ ] All content uses realistic mock data and no placeholder text.
