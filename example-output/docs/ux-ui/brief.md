# Product Brief

## Product name
Local & Cloud Network Audiobook Player for Android TV

## Problem statement
Android TV media players are optimized for video or short-form music, not 40-hour audiobook sessions. Home server and cloud storage listeners need exact progress persistence, reliable remote streaming, and D-pad-first browsing that does not block on live SMB or cloud folder traversal.

## Product purpose
The product is a native Android TV audiobook player that indexes local network and cloud libraries into a local metadata cache, then streams audio on demand from provider-backed playable asset references. It gives living-room listeners fast browsing, exact resume, TV-safe playback controls, and source-specific repair flows.

## Target users
- Living-room listeners who resume and control long-form audiobooks from a TV remote.
- Home server owners who store audiobooks on SMB2/SMB3 shares.
- Cloud library owners who store audiobooks in Dropbox, Google Drive, iCloud Drive, or OneDrive.
- Internal/background systems: Sync Worker, Provider API, Playback Engine.

## Business goal
Deliver a lean Phase 1 Android TV prototype that proves cache-backed audiobook browsing, first-class source setup, remote streaming, millisecond resume, and resilient source repair before expanding into downloads or metadata enrichment.

## Success criteria
- Library grid renders cached catalog details in under 100ms after wake.
- Audio stream starts within 2000ms over typical 5GHz home Wi-Fi.
- Every interactive surface is D-pad navigable with overt focus state and 48dp to 96dp safe margins.
- Playback progress saves every 5 seconds and immediately on pause, skip, or termination.
- SMB and cloud failures explain source-specific recovery without losing cached context.

## Scope

### In scope
- SMB2/SMB3 source setup with username/password or anonymous login.
- First-class Dropbox, Google Drive, iCloud Drive, and OneDrive setup and authorization flows.
- Optional advanced WebDAV source path.
- Provider capability model for listing, ranged streaming, silent refresh, and resumability.
- Background indexing into Room/local metadata cache.
- Cached Home, Library, Book Detail, Now Playing, and Source Repair experiences.
- Media3/ExoPlayer playback with MediaSession foreground behavior.

### Out of scope
- Local on-device downloading.
- Third-party metadata scraping.
- Social features, reviews, recommendations, or account management.
- Treating WebDAV as the implementation path for first-class cloud providers.
- Mobile companion app implementation.

## Known constraints
- Native Android TV target, minimum SDK 26.
- Kotlin-centric stack: Compose for TV, Media3, Room, WorkManager, Coroutines/Flow, Hilt, jCIFS-NG, provider-specific APIs.
- Remote storage latency must not affect cached browsing screens.
- Credentials, IPs, tokens, refresh tokens, app passwords, session references, and provider account metadata require encrypted native storage.
- UI must be legible and operable from a 10-foot couch context.

## Key assumptions
- Phase 1 should prioritize source setup, library browsing, playback, and repair over source management administration.
- Household listener, SMB owner, and cloud owner may be the same person, but separate actors make flow ownership clearer.
- Provider authorization can be simulated in a prototype while preserving provider-specific states and recovery.
- Final brand tokens are not supplied; design defaults are provisional.
